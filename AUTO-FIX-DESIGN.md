# CodeSheriff Auto-Fix System — Design Document

_Author: Rache (Vish's AI engineer)_  
_Date: April 2026_  
_Status: Phase 1 implemented_

---

## Why This Exists

Cubic flags issues. We fix them.

That's the long-term wedge. Their auto-fix is a static lookup table — same canned fix for every SQL injection regardless of context. Ours learns from the team: what fixes developers actually accept, reject, and modify. Over time, our fix suggestions get better for each specific codebase. Theirs don't.

This is the data flywheel that makes CodeSheriff defensible at scale.

---

## Architecture Overview

```
Finding detected
     │
     ▼
AutoFixGenerator (Phase 1)
     │  Uses Claude + file context
     │  Returns: suggestedCode, explanation, confidence
     ▼
GitHub PR comment
     │  GitHub suggestion format → "Commit suggestion" button
     │  Developer clicks → fix applied with one click
     ▼
FixOutcome tracker (Phase 2)
     │  Webhook: PR review submitted / new commit
     │  Records: ACCEPTED | REJECTED | MODIFIED | IGNORED
     ▼
Confidence aggregator (Phase 3)
     │  Per (ruleId, pattern) — rolling acceptance rate
     │  Threshold: >90% acceptance rate
     ▼
Auto-commit mode (Phase 3)
     │  High-confidence fixes posted without waiting for manual review
     │  Team can toggle off per-rule or globally
```

## Phase 1: Generate Fixes (Implemented)

### What it does

After the ExplanationEngine enriches findings, a new **AutoFixGenerator** stage runs on all `HIGH` and `CRITICAL` findings. It:

1. Extracts the code at the finding's line range (±5 lines of context)
2. Calls Claude with the finding context + surrounding code
3. Returns a concrete code fix — the exact replacement lines
4. Embeds the fix in the inline PR comment as a GitHub suggestion block

### GitHub Suggestion Format

GitHub renders a code block with ` ```suggestion ` as a diff. The developer sees a "Commit suggestion" button. One click applies the fix. Zero friction.

```markdown
⚠️ **CodeSheriff: JWT decoded without verification**

This token is decoded client-side only — the signature is never verified. 
An attacker can forge any payload.

```suggestion
const decoded = jwt.verify(token, process.env.JWT_SECRET, { algorithms: ['HS256'] });
```

> 🤖 Auto-fix generated by CodeSheriff
```

### What AutoFixGenerator produces

```typescript
interface AutoFix {
  suggestedCode: string;   // replacement lines (no markdown, raw code)
  explanation: string;     // 1-2 sentences on what changed
  confidence: number;      // 0.0–1.0 (LLM self-assessment)
  lineStart: number;       // line range the fix applies to
  lineEnd: number;
}
```

### Scope limits

- Only `HIGH` and `CRITICAL` findings (controls LLM cost)
- Max 10 fixes per scan (same cap as inline comments)
- Only for PR scans, not push scans (no PR to comment on)
- Non-fatal: if fix generation fails, finding is still posted without suggestion

### New pipeline stage

Stage order after Phase 1:

```
1. AIPatternDetector
2. SecretsScanner
3. StaticAnalyzer
4. HallucinationDetector (concurrent)
5. AuthFlowValidator (concurrent)
6. LogicBugDetector (concurrent)
7. ExplanationEngine
8. SeverityScorer
9. AutoFixGenerator ← NEW (runs after scoring, so it knows severity)
```

### Feature flag

`enableAutoFix` in `AnalysisFeatureFlags`. Off for FREE tier (LLM cost), on for TEAM and ENTERPRISE.

## Phase 2: Track Outcomes (Post-Launch)

### Trigger

When GitHub sends a `pull_request_review` webhook with action `submitted`, or when a new commit pushes to the PR branch, we correlate against open findings.

### FixOutcome model

```prisma
model FixOutcome {
  id        String @id @default(cuid())
  findingId String
  finding   Finding @relation(...)
  
  suggestedFix   String @db.Text  // what we suggested
  actualFix      String? @db.Text // what the dev actually wrote (if different)
  outcome        FixOutcomeStatus
  
  // Metadata for the corpus
  ruleId         String?
  patternHash    String   // hash of the code pattern, not the specific code
  language       String
  
  acceptedAt  DateTime?
  rejectedAt  DateTime?
  createdAt   DateTime @default(now())
  
  @@index([ruleId])
  @@index([patternHash])
  @@index([outcome])
}

enum FixOutcomeStatus {
  ACCEPTED   // dev clicked "Commit suggestion"
  REJECTED   // dev pushed different code to fix the same line
  MODIFIED   // dev accepted with edits
  IGNORED    // finding closed but fix not used
  PENDING    // awaiting resolution
}
```

### Acceptance detection

GitHub doesn't give us a direct signal when "Commit suggestion" is clicked — it just shows up as a commit. We detect it by:

1. New commit pushes to the PR branch
2. We re-scan that file
3. If the suggested line is no longer flagged AND the new code matches our suggestion → `ACCEPTED`
4. If the flagged line is fixed but with different code → `MODIFIED` or `REJECTED`
5. If the finding is still there → `IGNORED`

### Corpus integration

Each `FixOutcome` with `ACCEPTED` or `MODIFIED` becomes a corpus entry:

```
{
  ruleId: "auth-jwt-client-only",
  issuePattern: (hash of original code pattern),
  acceptedFix: (the code that was committed),
  language: "typescript",
  weight: 1.0 (ACCEPTED) | 0.7 (MODIFIED)
}
```

This corpus feeds back into the autotune evolution loop — the fix generator gets better at producing fixes for patterns where it has seen what works.

## Phase 3: Auto-Commit High-Confidence Fixes (Later)

### Trigger condition

For a given `(ruleId, patternHash)`:
- At least 10 outcomes recorded (sample size floor)
- Rolling acceptance rate (last 30 outcomes) > 90%

### Auto-commit behavior

- Fix is posted to the PR immediately when the finding is detected
- No "Commit suggestion" step needed — we post it directly via GitHub API
- A banner is added to the PR summary: _"X fixes auto-applied by CodeSheriff (based on Y historical acceptances)"_
- Org can toggle auto-commit off globally or per-rule in the Settings page

### Safety gates

- Auto-commit NEVER runs on secrets findings (too high risk of a wrong fix hiding a real credential)
- Auto-commit requires TEAM or ENTERPRISE plan
- Auto-commit posts as a suggestion (not a direct commit) — dev still sees it and can revert
- Rate limit: max 5 auto-applied fixes per PR

### Confidence calculation

```typescript
function getFixConfidence(ruleId: string, patternHash: string): number {
  const recent = getRecentOutcomes(ruleId, patternHash, 30);
  if (recent.length < 10) return 0; // not enough data
  
  const accepted = recent.filter(o => 
    o.outcome === 'ACCEPTED' || o.outcome === 'MODIFIED'
  ).length;
  
  return accepted / recent.length;
}
```

## Why This Beats Cubic Long-Term

| | Cubic | CodeSheriff |
|---|---|---|
| Fix generation | Static lookup table | Claude with full code context |
| Fix quality over time | Static | Improves from real acceptance data |
| Team-specific patterns | ❌ | ✅ (learns per repo/org) |
| Fix corpus | None | Grows with every accepted fix |
| Auto-commit | ❌ | ✅ (Phase 3, confidence-gated) |

The static approach caps out quickly — you can only hardcode so many fix patterns. Our approach has no ceiling: every team that uses CodeSheriff makes it smarter for the next team.

## Open Questions

1. **Suggestion line range**: GitHub suggestions apply to the exact lines of the inline comment. If the fix changes the number of lines, we need to set `start_line` + `line` carefully. Current impl uses `lineStart`/`lineEnd` from the finding.

2. **Secrets findings**: Should we generate fixes for secrets? (e.g., "replace with env var"). Decision: yes, but mark confidence = 0 so they never auto-commit in Phase 3.

3. **Multi-file fixes**: Some bugs require changes to multiple files. Phase 1 only handles single-file suggestions. Multi-file is Phase 3 territory.

4. **Prompt injection in fix output**: The fix generator sees user code. Output is sanitized (it's just code, not eval'd), but we should validate that the output looks like code before posting.
